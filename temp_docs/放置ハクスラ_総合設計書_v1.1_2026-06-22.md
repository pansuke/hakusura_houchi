# 1. Vision

本作は以下を組み合わせたゲームである。

* 放置ゲーム
* ハクスラ
* キャラ収集
* カード収集
* デッキ構築
* オートチェス
* LoL的なレーン戦
* ポケモン的な選出・相性・愛着
* 将来的なPvP

ゲームの主軸:

```text
Character
↓
Deck
↓
Equipment
↓
Build
↓
Party
↓
Formation
↓
Battle
```

プレイヤーは戦闘中に直接操作しない。

構築・選出・配置によって戦闘結果が変化することを目指す。

---

# 2. 最重要設計原則

1. カードを戦闘の主役にする
2. 通常攻撃を主役にしない
3. 距離・座標管理を行わない
4. BattleEngineとViewerを分離する
5. 戦闘はAction Indexで進行する
6. 表示時間を戦闘計算から分離する
7. 同一入力・同一Seedで同一結果を生成する
8. Masterデータは個別JSONを正本とする
9. 統合JSONを直接編集しない
10. 未確定仕様を推測で実装しない
11. Master・Instance・構築データ・BattleStateを分離する
12. ゲームロジックをFastAPIやViewerへ依存させない

# 4. Masterデータ基盤

## 4.1 正本

Masterデータの正本は、`data/source`配下の個別JSONとする。

```text
data/
  source/
    characters/
    traits/
    cards/
  generated/
    game-data.json
```

統合済みの`game-data.json`を直接編集しない。

## 4.2 1レコード1JSON

```text
data/source/characters/character_warrior_001.json
data/source/traits/trait_adjacent_splash.json
data/source/cards/card_fire_ball.json
```

1種類のMasterを巨大JSONへまとめない。

1キャラクターを複数JSONへ細分化する方式もM0-Bでは採用しない。

## 4.3 M0-B対象Master

M0-Bでは以下のみを対象とする。

* CharacterMaster
* TraitMaster
* CardMaster

以下は後続で追加する。

* ItemMaster
* StageMaster
* AreaMaster
* DropTableMaster
* StatusEffectMaster
* EnemyFormationMaster

目的はすべてのMasterを完成させることではなく、Master追加可能な基盤を完成させることである。

# 5. ID設計

## 5.1 ID形式

* 小文字snake_case
* Master種別プレフィックス
* ファイル名とIDを一致
* 外部参照へ使用
* 原則変更しない

例:

```text
character_warrior_001
trait_adjacent_splash
card_fire_ball
```

形式:

```regex
^[a-z][a-z0-9_]*$
```

## 5.2 ファイル名

```text
<id>.json
```

例:

```text
card_fire_ball.json
```

データ内IDが`card_fire_ball`でない場合はエラーとする。

# 6. JSON Schema

## 6.1 正本

M0-BではJSON SchemaをMaster構造の正本とする。

```text
schemas/
  character.schema.json
  trait.schema.json
  card.schema.json
  definitions/
    effect.schema.json
    roll_rule.schema.json
```

同一構造を表すPydanticモデルを手作業で二重管理しない。

## 6.2 JSON Schemaを選ぶ理由

* JSONを直接検証できる
* Python以外からも利用できる
* Viewerや将来のツールでも再利用可能
* Pydanticとの同期ずれを避けられる
* Master定義とAPIモデルを分離できる

Pythonランタイムモデルは、BattleEngineが統合データを読み込む段階で追加する。

# 7. build_data.py

## 7.1 役割

`build_data.py`は以下を行う。

1. 個別JSONの列挙
2. JSON構文検証
3. JSON Schema検証
4. ID形式検証
5. ファイル名とIDの一致検証
6. ID重複検証
7. Master間参照検証
8. ID順ソート
9. 統合JSON生成
10. 一時ファイル書込
11. 成功時のみ正式ファイルへ置換

## 7.2 入力

```text
data/source/**/*.json
schemas/**/*.schema.json
```

## 7.3 出力

```text
data/generated/game-data.json
```

例:

```json
{
  "schema_version": 1,
  "characters": [],
  "traits": [],
  "cards": []
}
```

## 7.4 実行時方針

BattleEngineやAPIは、原則として個別JSONを直接読まない。

検証済みの`game-data.json`を入力とする。

# 8. 整合性チェック

## 8.1 JSON構文

* JSONとして読める
* 可能であれば重複キーを拒否する

## 8.2 Schema

* 必須項目
* 型
* enum
* 文字列形式
* 数値範囲
* 配列件数
* 未定義フィールド

## 8.3 ID

* ID形式
* ID重複
* ファイル名一致

## 8.4 参照

例:

```text
CharacterMaster.trait_ids
```

が実在するTraitMasterを指すこと。

## 8.5 数値制約

例:

* HP > 0
* MP >= 0
* MPコスト >= 0
* RollRule.min <= RollRule.max
* weighted_table.weight > 0

## 8.6 対象外

以下はM0-Bの整合性チェック対象外。

* キャラ間バランス
* カードのコスト効率
* Legendaryの強さ
* 戦闘が終了するか
* Deckが正常に回るか
* ステージ攻略可能性

# 9. エラー設計

検証失敗時は、可能な限り全エラーを列挙する。

例:

```text
ERROR data/source/characters/character_warrior_001.json
  $.trait_ids[0]
  Referenced TraitMaster "trait_missing" does not exist.

ERROR data/source/cards/card_fire_ball.json
  $.mp_cost
  Value must be greater than or equal to 0.
```

終了コード:

```text
0: 成功
1: 失敗
```

失敗時:

* 既存の正常な`game-data.json`を破壊しない
* 不完全な生成物を残さない

# 10. 再現可能ビルド

同一入力から、バイト単位で同一の生成物を作る。

生成物へ含めないもの:

* 実行日時
* マシン名
* 絶対パス
* ランダムID
* OS依存の列挙順

固定するもの:

* UTF-8
* LF
* インデント
* キー順
* 配列順
* 最終改行

---

# 11. 生成物のGit管理

M0-Bでは以下をGit管理する。

* `data/source`
* `schemas`
* `data/generated/game-data.json`

理由:

* Pull Requestで差分を確認できる
* Master変更の影響をレビューできる
* 生成再現性を確認できる
* 開発初期の調査が容易

将来データ量が大きくなった場合、生成物をGit管理しない方式を再検討する。

# 14. Action方式

## 14.1 Action

戦闘中の最小進行単位。

```text
Action開始
↓
現在行動者決定
↓
Gauge更新
↓
カード試行
↓
Effect解決
↓
死亡・終了判定
↓
次行動者決定
↓
Action終了
```

## 14.2 ActionRight

演出上は行動権がキャラクター間を移動する。

ドメイン上では、BattleStateが現在行動者を一意に保持するか、Action Queueの先頭を現在行動者とする。

独立した「権利オブジェクト」を作るかは、Action Scheduler設計時に決定する。

## 14.3 Action Index

Battle内の通し番号。

用途:

* Event
* Snapshot
* Replay
* Viewer位置
* デバッグ

# 15. Gauge

DS・MRG・HRGはAction進行型Gaugeとして扱う。

```text
currentValue += gainPerTrigger
```

しきい値到達時に処理を発動する。

超過分は繰り越す。

共通Gauge構造:

* currentValue
* gainPerTrigger
* threshold
* maxValue候補
* overflow
* triggerCount

---

# 16. Viewer

Viewerは戦闘計算を行わない。

BattleEngineが生成したイベントとSnapshotを表示する。

開発Viewer機能:

* 次の行動
* 前の行動
* 10行動進む
* 100行動進む
* 最初へ戻る
* 最後まで進む
* 任意Actionへ移動
* 自動再生
* 表示速度変更
* HP / MP / Gauge
* 手札
* 保留カード
* 失敗理由
* Event詳細

表示時間は演出上の値であり、戦闘結果へ影響しない。

# 17. カード設計

## 17.1 CardMaster

カードそのものの定義。

* id
* name
* rarity
* mp_cost
* effects
* roll_rules
* deck_rules

## 17.2 CardInstance

取得したカード個体。

* instance_id
* master_id
* rolled_values
* owner
* lock
* favorite

## 17.3 レアリティ

CardMaster自体の入手しにくさ。

高レアリティほどドロップしにくい。

Legendaryは意図的にコスト効率が高い強カードとして設計可能。

## 17.4 性能ロール

同じCardMasterを取得した際の個体性能差。

レアリティと性能ロールは分離する。

## 17.5 RollRule

* fixed
* flat_range
* percent_range
* weighted_table

`tier_table`は使用しない。

離散値抽選は`weighted_table`へ統一する。

---

# 18. Deck構築制限

Legendaryは意図的に強いカードとなるため、MP不足だけでは採用抑制にならない可能性がある。

候補:

* Legendary総数上限
* CardMasterごとの最大採用枚数
* PvP専用制限
* DeckRule

制限はコードへ埋め込まず、データ定義可能にする。
