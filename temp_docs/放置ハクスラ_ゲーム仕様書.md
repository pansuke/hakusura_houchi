# 3. 探索システム

## 3.1 1探索

1探索は、本作における放置単位である。

- 1探索は最大約2時間を想定
- 探索中に複数ステージを連続攻略
- 敗北すると探索終了
- 探索結果に応じて報酬を獲得

## 3.2 探索深度

探索が進むほど探索深度が上がる。

探索深度は報酬に影響するが、ボーナスは逓減させる。

敗北時の候補:

- 探索終了
- 探索深度リセット
- 取得資源の一部ロスト

# 5. 戦闘システム

## 5.1 基本方針

- カード主体
- カードは自動で試行・発動される
- キャラごとにDeckを持つ
- プレイヤーは戦闘中に操作しない
- 通常攻撃を主役にしない
- 2D座標、自由移動、射程距離などの詳細管理は行わない
- M3以降はレーンごとの抽象化された1次元位置だけを扱う
- 複雑なAI判断は使用しない
- BattleEngineを戦闘状態と行動解決の唯一の正とする
- ViewerはBattleEngineが生成したEventとSnapshotを表示するだけとする

## 5.2 Action方式

戦闘は実時間ではなく、Action単位で進行する。

Actionとは、現在行動者へ制御が渡ってから、その行動結果と戦闘終了判定が確定するまでの最小進行単位である。

```text
Action開始
↓
現在行動者の確定
↓
ゲージ更新
↓
カード選択・試行
↓
Effect解決
↓
死亡判定
↓
戦闘終了判定
↓
Action終了
↓
次の生存キャラクターへ移行
```

Action IndexはBattle内の通し番号である。

```text
Action #1
Action #2
Action #3
```

表示上の演出時間はViewer側で管理する。

演出速度を変更しても、戦闘結果は変化しない。

## 5.3 M1の目的

M1では、1レーンに配置された固定の味方1体と敵1体による戦闘を対象とする。

M1の目的は、最終的な戦闘バランスを完成させることではなく、以下を検証可能にすることである。

- 現在行動者が常に一人だけ決まっている
- 1Actionずつ状態を更新できる
- カードを自動で試行・発動できる
- HPとMPが変化する
- Actionごとの構造化Eventを生成できる
- 死亡と戦闘終了を判定できる
- 同じ初期状態とSeedから同じ結果を再現できる
- ViewerでAction単位に再生できる

## 5.4 M1の対象外

M1では以下を実装しない。

- 3レーン
- SUPPORT
- 進軍
- ネクサス
- 速度ステータスによる行動順計算
- 追加Action
- 追撃
- 反撃
- 割り込み
- 同時行動
- 状態異常による行動不能
- 復活
- CardInstanceの性能ロール
- 装備
- PvE探索
- PvP

## 5.5 M1の戦闘開始状態

M1のBattleScenarioは、固定された以下のデータを入力として持つ。

- 味方CharacterMaster ID
- 敵CharacterMaster ID
- 味方Deck
- 敵Deck
- 固定行動順
- Battle Seed
- 最大Action数
- 各参加者の初期HP / MP
- 各参加者のDS / MPR / HPR

M1ではキャラクター配置、前衛・後衛、距離、射程を扱わない。

HP、MP、DS、MPR、HPRなどの初期値はCharacterMasterおよびBattleScenarioから生成する。

装備、育成値、スキルツリーはM1の入力に含めない。

## 5.6 M1の行動順

M1では行動順をBattleScenarioの固定配列として与える。

例:

```text
character_ally_001
character_enemy_001
```

BattleStateは内部的に以下を保持する。

- `turn_order`
- `current_turn_index`
- `action_index`
- 参加者ごとのRuntime State

Snapshotでは、表示上の行動者を以下の2項目へ分離する。

- `acted_actor_id`: そのActionで行動したキャラクター
- `next_actor_id`: 次に行動予定の生存キャラクター

戦闘開始時の現在行動者は`turn_order[0]`とする。

Action完了後は、配列上の次の生存キャラクターを次行動者とする。

配列の末尾へ到達した場合は先頭へ戻る。

M1では以下を行わない。

- 速度値による順番計算
- 行動速度ゲージ
- 同値判定
- ランダムな先攻決定
- 同一キャラクターの連続行動

固定行動順はM1専用の暫定仕様であり、3レーン対応前のAction Scheduler設計で再検討する。

## 5.7 M1における1Actionの境界

1Actionには以下を含む。

1. 次の生存キャラクターを現在行動者として確定する
2. `action_index`を1進める
3. `action_started`を生成する
4. 現在行動者のHPRを解決する
5. 現在行動者のMPRを解決する
6. 現在行動者のDSをDraw Gaugeへ加算する
7. Draw Gaugeがしきい値へ到達した回数分だけドローを試行する
8. 現在行動者が使用を試みるカードを手札先頭から決定する
9. MP、対象、発動条件を検証する
10. 発動可能な場合はMPを消費し、`card_used`を生成する
11. カードに定義されたEffectを記載順に解決する
12. 各Effect解決後に死亡判定を行う
13. カード内のEffect解決後に戦闘終了判定を行う
14. 戦闘継続時のみ次の生存キャラクターを決定する
15. `action_completed`を生成する
16. 勝敗確定時は`battle_completed`を生成する
17. Action完了後Snapshotを保存する

1枚のカードに複数Effectがある場合、それらを同一Action内で記載順にすべて解決する。

M1ではAction途中に別キャラクターのActionを割り込ませない。

## 5.8 M1のHPR / MPR / DS

HPR、MPR、DSはいずれも行動権を獲得した現在行動者にだけ適用する。

相手のAction中、行動権を持っていない側はHP回復、MP回復、Draw Gauge進行を行わない。

HPRとMPRはGaugeではなく、行動権獲得時の確定回復である。

```text
Action開始
↓
現在行動者のHPRをHPへ加算
↓
現在行動者のMPRをMPへ加算
↓
現在行動者のDSをDraw Gaugeへ加算
↓
しきい値到達回数分だけドロー試行
↓
現在行動者のカード処理
```

HPとMPは最大値を超えない。

```text
hp = min(max_hp, hp + hpr)
mp = min(max_mp, mp + mpr)
draw_gauge += ds
```

Draw Gaugeのしきい値は100とする。

しきい値以上になった場合は到達回数分だけドロー権を得て、超過分を繰り越す。

```text
trigger_count = draw_gauge // 100
draw_gauge = draw_gauge % 100
```

1回の加算で複数回しきい値へ到達した場合は、到達回数分だけ処理する。

死亡キャラクターは行動権を得ないため、HPR、MPR、DSを処理しない。

## 5.9 M1のカード選択

M1ではプレイヤーによるカード選択を行わない。

現在行動者は、手札を先頭から確認し、最初に発動可能なカードを使用する。

発動可能条件:

- 必要MP以上を持つ
- 生存している有効対象が存在する
- カード固有の追加条件を満たす

発動可能なカードがない場合、そのActionではカードを発動しない。

MP不足カードは捨てず、手札に保持する。

MP不足などで発動できないカードは`card_held`として記録する。

M1ではカードの優先度AI、評価関数、ランダム選択を実装しない。

## 5.10 M1の対象選択

M1は味方1体・敵1体のため、敵対象カードは生存している唯一の敵を対象とする。

味方対象カードは使用者自身を対象とする。

対象が存在しない場合、そのカードは発動できない。

対象選択に乱数は使用しない。

## 5.11 M1のダメージ

M1では、Damage Effectに定義された固定値をそのままHPから減算する。

```text
damage = effect.value
remaining_hp = max(0, current_hp - damage)
```

`damage_applied`は、解決前後を比較できるよう以下をPayloadへ含める。

- `before`
- `requested`
- `applied`
- `after`

M1では以下を計算しない。

- AD / AP倍率
- AR / MR軽減
- CRT
- EVA
- 属性相性
- ダメージ乱数
- 最低ダメージ補正

本番用ダメージ式は後続マイルストーンで定義する。

## 5.12 行動不能・死亡時の扱い

M1では状態異常による行動不能を実装しない。

死亡条件:

- HPが0になったキャラクターを死亡状態とする
- 死亡キャラクターはカードを使用できない
- 死亡キャラクターはカード対象にならない
- 死亡キャラクターは行動権を得ない
- 死亡キャラクターの順番へ到達した場合はAction Indexを消費せずスキップする

M1では復活を実装しない。

1対1戦闘のため、一方が死亡した時点で戦闘終了条件を満たす。

## 5.13 M1の戦闘終了条件

各Effect解決後に死亡判定を行う。

現在のカードに含まれるすべてのEffectを解決した後、以下の順で戦闘終了を判定する。

1. 味方と敵の双方が全滅している場合は引き分け
2. 敵が全滅している場合は味方勝利
3. 味方が全滅している場合は味方敗北
4. 最大Action数へ到達した場合は引き分け
5. 上記以外は戦闘継続

勝敗確定後:

- BattleStatusを`completed`へ変更する
- BattleResultを設定する
- `battle_completed`を生成する
- 次の現在行動者を決定しない
- 新しいActionを開始しない

M1の最大Action数はBattleScenarioで指定し、初期値は1000とする。

## 5.14 BattleStateの正

BattleStateの唯一の正はPythonのBattleEngineとする。

FastAPIはBattleEngineを呼び出すAPI層であり、戦闘ルールを保持しない。

ViewerはBattleStateを計算・変更しない。

M1ではBattleEngineが戦闘開始から終了までを一括計算し、以下を生成する。

- `events`
- `snapshots`
- `summary`
- `display_catalog`

`display_catalog`はViewer表示用の名前・説明を提供する補助情報であり、戦闘計算の正ではない。

BattleEngineは表示言語やMaster名称を保持しない。

API層はMasterデータから`display_catalog`を生成し、未登録IDは通常画面に露出させず`名称未設定`として扱う。

「次の行動を見る」は、Backendへ次のAction実行を要求する操作ではない。

Viewerが、計算済みReplayの表示位置を1Action進める操作とする。

M1ではBattleStateをDBまたはサーバーメモリへ永続保存しない。

ページ再読み込み時の復元、複数タブ操作、APIの二重送信制御はM1の対象外とする。

## 5.15 BattleEventの最小形式

画面表示用文字列ではなく、構造化されたBattleEventを正とする。

すべてのBattleEventは以下を持つ。

```json
{
  "event_id": 1,
  "action_index": 1,
  "sequence": 1,
  "event_type": "action_started",
  "actor_id": "ally_001",
  "target_id": null,
  "payload": {}
}
```

必須項目:

- `event_id`: Battle内のイベント通し番号
- `action_index`: 所属するAction Index。Battle開始イベントは0
- `sequence`: 同一Action内の表示順
- `event_type`: イベント種別
- `actor_id`: 発生主体。存在しない場合はnull
- `target_id`: 対象。存在しない場合はnull
- `payload`: イベント固有データ

M1で使用するEvent Type:

- `battle_started`
- `action_started`
- `gauge_changed`
- `card_drawn`
- `card_draw_blocked`
- `card_attempted`
- `card_held`
- `card_used`
- `mana_recovered`
- `mana_gained`
- `mana_spent`
- `health_recovered`
- `damage_applied`
- `character_defeated`
- `action_completed`
- `battle_completed`

Action権によるMPRは`mana_recovered`、カードEffectによるMP獲得は`mana_gained`として分離する。

`card_drawn`と`card_draw_blocked`は、Draw Gauge由来かカードEffect由来かを判別するため`draw_source`をPayloadへ持つ。

`card_draw_blocked`は以下をPayloadへ持つ。

- `blocked_reason`: `hand_full`または`empty_deck`
- `draw_source`: `draw_gauge`または`card_effect`
- `hand_size`
- `hand_limit`: `hand_full`時のみ

表示用の日本語コメントはViewer側でEventと`display_catalog`から生成する。

## 5.16 BattleSnapshotの最小形式

Viewerが任意のAction状態を表示できるよう、M1ではAction完了後のBattleStateをSnapshotとして保存する。

最低限保持する項目:

- `action_index`
- `battle_status`
- `battle_result`
- `acted_actor_id`
- `next_actor_id`
- 各参加者の`participant_id`
- 各参加者の`character_master_id`
- 各参加者の`side`
- 各参加者のHP / 最大HP
- 各参加者のMP / 最大MP
- 各参加者の生存状態
- 各参加者のDS / MPR / HPR
- 各参加者のDraw Gauge
- 各参加者の手札
- 各参加者の山札
- 各参加者の捨て札

Action #0として戦闘開始直前の初期Snapshotも保存する。

## 5.17 開発Viewerの操作

開発Viewerでは計算済みReplayをAction単位で確認する。

- 次の行動
- 前の行動
- 10行動進む
- 100行動進む
- 最後まで進む
- 最初へ戻る
- 任意Actionへ移動
- 自動再生
- 表示速度変更

「次の行動」を1回押すごとに、表示位置を1Actionだけ進める。

演出中は操作ボタンを無効化し、多重表示処理を防止する。

戦闘終了後は「次の行動」を無効化する。

表示速度と演出時間は戦闘計算へ影響しない。

M2以降の開発Viewerは、Action Summaryを以下のフェーズで表示する。

1. 行動準備
2. ドロー
3. カードアクション
4. 効果解決
5. 行動終了

通常表示では日本語名を優先し、内部IDや生EventはDebug表示に隔離する。

## 5.18 M1の再現性

M1では、同一の以下を入力した場合、同一のEvent、Snapshot、Summaryを生成する。

- `game-data.json`
- BattleScenario
- Deck

M1のカード選択、対象選択、固定ダメージには乱数を使用しない。

山札シャッフルをM1で実施する場合は、必ずBattle Seedから生成したBattle専用乱数器を使用する。

## 5.20 M3本番戦闘仕様

M3では、M1の1対1・固定値・暫定Deck Runtimeを、本番用の3レーン戦闘へ拡張する。

M3は以下へ分割する。

- M3-A: 3レーン、本番Deck、位置・PUSH・対面、復活、本番Damage、Local / Adjacent / Global
- M3-B: Nexus、Nexus攻撃、本番勝敗

SUPPORTはM4-Aで追加する。

### 5.20.1 3レーンとAction Scheduler

戦闘レーンは以下の3本とする。

```text
TOP
MID
BOT
```

M3の基本配置は、各レーンにつき味方1体・敵1体とする。

Battle全体で単一のAction Indexを持ち、行動機会は以下の固定順で巡回する。

```text
TOP味方
TOP敵
MID味方
MID敵
BOT味方
BOT敵
↓
繰り返し
```

死亡中のキャラクターにも行動機会は到来する。死亡中の行動機会は復活待ちカウントを進めるため、Action Indexを消費する。

速度ステータスによる行動順変更、同時行動、割り込みはM3では扱わない。

### 5.20.2 M3のAction Pipeline

M3の1Actionは以下の順で処理する。

1. Schedulerが現在行動者と所属レーンを確定する
2. `action_index`を1進める
3. `action_started`を生成する
4. 死亡中の場合は復活待ちを処理する
5. 復活条件を満たした場合は復活・Deck再初期化を行う
6. 移動・PUSH・対面判定を行う
7. 対面または敵Nexus到達状態を確定する
8. カード行動可能状態ならHPR / MPR / DSを処理する
9. Draw Gauge発火回数分のドローを処理する
10. 最古の手札だけを使用判定する
11. 使用可能ならカードを使用し、Effectを記載順に解決する
12. `consumes_action=false`の場合のみ次の最古カードを続けて判定する
13. 死亡・Nexus破壊を判定する
14. `action_completed`を生成する
15. Nexus破壊時は`battle_completed`を生成する
16. Snapshotを保存する

対面相手または攻撃可能な敵Nexusへ到達していないキャラクターは、移動だけでActionを終了する。この場合、HPR / MPR / DS、ドロー、カード使用は行わない。

同じActionの移動で相手へ到達した場合は、そのAction内でHPR / MPR / DS、ドロー、カード使用まで続行できる。

### 5.20.3 Battle Rule Config

調整値をコードへ埋め込まない。

以下をBattle Rule Configとして管理し、開発Viewerの設定画面から編集・保存可能にする。

| 設定 | 初期値 |
|---|---:|
| 初期手札枚数 | 3 |
| 最大手札枚数 | 7 |
| Draw Gaugeしきい値 | 100 |
| 復活で失う自分の行動機会 | 3 |
| 味方Nexus位置 | -1000 |
| 敵Nexus位置 | 1000 |
| 初期対面位置 | 0 |
| Nexus最大HP | 8000 |
| Nexus AR | 0 |
| Nexus MR | 0 |
| 防御計算定数 | 100 |

Frontendは設定の入力・保存UIを提供するが、戦闘ルールの正にはならない。

```text
開発Viewer
↓ GET / PUT
FastAPI
↓
Battle Rule Config
↓
BattleEngine
```

使用したBattle Rule ConfigはReplayへ含め、同一Config・同一入力・同一Seedから同一Replayを再現できるようにする。

### 5.20.4 本番Deck Runtime

戦闘開始時はBattle Seedを用いてDeckを必ずシャッフルする。

```text
DeckをSeed付きシャッフル
↓
先頭3枚を手札へ移動
↓
残りを山札とする
```

- 初期手札: 3枚
- 最大手札: 7枚
- 使用後のカード: 捨て札の末尾
- 山札が空になった場合: 捨て札をSeed付きでシャッフルして山札へ戻す
- 同一入力・同一Seedでは同じシャッフル結果にする

ドロー時に手札が最大枚数へ達している場合、最も古い手札を捨て札へ移動した後、新しいカードを手札末尾へ追加する。

```text
手札先頭 = 最も古いカード
手札末尾 = 最も新しいカード
```

最大手札時のドローは失敗ではない。

```text
最古カードを捨てる
↓
新しいカードを引く
```

#### カード使用順

手札の最も古いカードだけを判定する。

- 使用可能なら使用する
- MP不足、対象なし、発動条件未達なら手札に保持する
- 使用不能でも後続カードは探索しない
- 次回以降のMPR回復または手札超過による破棄を待つ

これにより、Deck順・Draw順・手札滞留がBuildの一部になる。

#### Action消費

カードは以下を持つ。

```text
consumes_action: true | false
```

- `true`: 使用後にカードアクションを終了する
- `false`: 使用後、新しく最古になったカードを続けて判定する

最古カードが使用不能になった時点でカードアクションを終了する。

無限連鎖は戦闘結果へ変換せず、Engineのシミュレーション安全上限超過エラーとして扱う。

### 5.20.5 乱数と再現性

Battle Seedから用途別の決定的乱数列を生成する。

- 戦闘開始Deckシャッフル
- 捨て札再シャッフル
- 復活時Deckシャッフル
- MIDのAdjacent対象選択
- 将来のCRT / EVA

異なる用途で同じ乱数列を直接共有しない。機能追加によって既存のDeck順が変わることを防ぐ。

### 5.20.6 個別位置とPUSH

各キャラクターは所属レーン内の個別位置を持つ。

```text
味方Nexus                         敵Nexus
-1000 ───────── 0 ───────── +1000
```

戦闘開始時は各レーンの味方・敵を位置0へ配置し、対面状態から開始する。

復活時は自陣Nexus位置から再出発する。

- 味方復活位置: -1000
- 敵復活位置: +1000

PUSHはキャラクターステータスであり、前線を押す能力と移動量を表す。

暫定的な目安:

- 低い: 10前後
- 標準: 20～30
- 高い: 40前後
- キャラクター内でも非常に高い: 50前後

#### 非対面時

行動者は敵側へPUSH分だけ移動する。

移動経路上に生存している対面候補がいる場合、その位置を越えず、相手位置で停止して対面状態になる。

同じActionで対面した場合は直後にカード行動できる。

#### 対面時

対面中は味方・敵が同じ位置を持つ。

行動者のPUSHが対面相手より高い場合だけ、その差分だけ両者を行動者の進軍方向へ移動させる。

```text
advance = max(0, actor.push - opponent.push)
```

例:

```text
味方PUSH 50
敵PUSH   30
味方Action時のadvance = 20
敵Action時のadvance   = 0
```

相手死亡中は対面相手のPUSHを0として扱うのではなく、対面を解除し、生存者は非対面移動として自身のPUSH全量で進む。

位置は-1000～1000へclampする。

### 5.20.7 復活

死亡したキャラクターはBattleから除外されず、自分の行動機会によって復活待ちを進める。

暫定値は、自分の行動機会を3回失うこととする。

```text
死亡時: respawn_turns_remaining = 3
次の自分の行動機会: 3 → 2、行動なし
次: 2 → 1、行動なし
次: 1 → 0、行動なし
次の行動機会: 復活して行動可能
```

復活待ちはBattle全体のAction数ではなく、そのキャラクター自身の行動機会だけで減少する。

復活時:

- HP: 最大
- MP: 最大
- 位置: 自陣Nexus
- 手札: 全破棄後、Deckを再構築して初期手札3枚
- 山札: 元DeckをSeed付きで再シャッフルした残り
- 捨て札: 空
- Draw Gauge: 0
- 一時Battle Effect: 解除

復活した行動機会では、そのまま前線方向へ移動する。

- 相手へ到達しなければ移動だけで終了
- 相手へ到達すれば、対面確定後にドロー・カード行動まで続行
- 敵Nexusへ到達し、対面相手がいなければNexusへ通常カードを使用可能

復活は死亡キャラクター自身のAction開始時にだけ発生する。別キャラクターのAction途中へ割り込まない。

### 5.20.8 本番Damage式

Damage Effectはカードごとに基礎値とScalingを持つ。

```text
Raw Damage
= base_damage
+ Σ（参照ステータス × ratio）
```

一枚のカードがADとAPの両方を参照することも許可する。

例:

```text
base_damage = 50
AD ratio = 0.6
AP ratio = 0.4
```

Damage Type:

- physical: ARで軽減
- magic: MRで軽減
- true: 防御軽減なし

防御値が0以上の場合の軽減式:

```text
physical_damage = raw_damage × 100 / (100 + AR)
magic_damage    = raw_damage × 100 / (100 + MR)
```

- 中間計算では丸めない
- 最終ダメージ確定時だけ切り捨てる
- Damage Effectが成立した場合の最低ダメージは1
- HPを0未満にしない

Ratioは浮動小数ではなく整数のbasis pointsで保持する。

```text
10000 = 1.0
8000  = 0.8
```

### 5.20.9 Effectの対象範囲

敵対象Effectは以下のScopeを持つ。

```text
local
adjacent
global
```

#### Local

自レーンを対象とする。

- 対面相手がいる: 対面キャラクター
- 対面相手がいない、敵Nexus未到達: 有効対象なし
- 対面相手がいない、敵Nexus到達済み: 敵Nexus

Nexusへ適用できるのはM3ではDamage Effectだけとする。

#### Adjacent

隣接レーンの生存敵キャラクター1体を対象とする。

- TOPからはMIDのみ
- BOTからはMIDのみ
- MIDからはTOP / BOTの有効対象からBattle Seedでランダム選択
- 有効な隣接対象が1体もいなければ不発
- 隣接レーン経由でNexusを攻撃しない

#### Global

TOP / MID / BOTにいる生存敵キャラクター全員を対象とする。

- 対象解決順はTOP → MID → BOT
- 敵キャラクターが1体もいなければ不発
- GlobalはNexusを対象にしない
- 自レーンを押し切った成果を別レーンやGlobal経由でNexus攻撃へ転用しない

`target=self`のEffectはM3では`scope=local`だけを許可する。

### 5.20.10 Nexus

各チームは共通のNexusを1つ持つ。

暫定値:

```text
Nexus Max HP = 8000
Nexus AR = 0
Nexus MR = 0
```

Nexusへ専用攻撃は使用しない。通常カードのDamage Effectを使用する。

Localの攻撃対象は以下で決定する。

```text
自レーンで対面中
→ 対面キャラクター

対面していない、敵Nexusへ未到達
→ 移動のみ

対面していない、敵Nexusへ到達
→ 敵Nexus
```

敵が自身の行動機会に復活し、移動後に対面状態になった場合、ドロー前に対面が確定する。そのAction以降、相手はNexusではなく復活したキャラクターを対象にする。

復活はAction途中へ割り込まないため、Nexus攻撃解決中に突然対象が切り替わることはない。

### 5.20.11 戦闘終了

本番の勝敗条件はNexus破壊だけとする。

```text
敵Nexus HP <= 0
→ 味方勝利

味方Nexus HP <= 0
→ 味方敗北
```

以下は本番の勝敗条件にしない。

- キャラクター全滅
- 一定Action数到達
- 一定時間経過
- 位置がNexusへ到達しただけ

本番ルールには最大Action数を設けない。Nexusが破壊されるまでBattleを継続する。

無限ループや実装不具合を検出するための`simulation_safety_limit`は開発・運用上の安全機構として別に持つ。到達時は引き分けではなくシミュレーションエラーとする。

# 17. 現時点の重要結論

- 本作はカード主体である
- プレイヤーのリアルタイム操作は行わない
- 戦闘はBattle全体のAction Indexで進行する
- M3の行動順はTOP味方 → TOP敵 → MID味方 → MID敵 → BOT味方 → BOT敵
- HPR / MPRは行動権獲得時の確定回復であり、DSだけがDraw Gaugeを進める
- M3では戦闘開始時とRecycle時にSeed付きシャッフルを行う
- 初期手札は3枚、最大手札は7枚
- 手札上限時のDrawは最古カードを捨てて新しいカードを入れる
- 最古カードだけを使用判定し、使用不能時は後続カードを探索しない
- カードがActionを消費するかは`consumes_action`で定義する
- 各キャラクターはレーン内の個別位置とPUSHを持つ
- 復活時はHP / MP最大、Deck・手札・Draw Gaugeをリセットし、自陣Nexusから再出発する
- 復活待ちは自分の行動機会を3回失う暫定値とする
- 本番Damageはカードの基礎値と標準化されたScalingで計算する
- AR / MR軽減は100 / (100 + defense)を用いる
- 中間計算では丸めず、最終ダメージだけ切り捨て、最低ダメージ1とする
- Localは自レーン、Adjacentは隣接する生存敵1体、Globalは全生存敵キャラクターを対象とする
- Adjacentに有効対象がなければ不発とし、Nexusへ転送しない
- GlobalはNexusを対象にしない
- Localは敵Nexusへ到達し、対面相手がいない場合だけ通常DamageカードでNexusを攻撃する
- Nexus最大HPは暫定8000

- 本番ルールに最大Action数はなく、安全上限到達は引き分けではなくエラーである
- 進軍専用カード分類は作らない
- Battle Rule Configは開発Viewerから編集・保存可能にする
- BattleStateの唯一の正はPythonのBattleEngineとする
- Viewerは戦闘計算を行わない
- 同一Config・同一入力・同一Seedから同一Replayを生成する
