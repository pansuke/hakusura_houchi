# 3. 探索システム

## 3.1 1探索

1探索は、本作における放置単位である。

- 1探索は最大約2時間を想定
- 探索中に複数ステージを連続攻略
- 敗北すると探索終了
- 探索結果に応じて報酬を獲得

## 3.2 探索深度

探索が進むほど探索深度が上がる。

探索深度は報酬に影響するが、ボーナスは逓減させる。

敗北時の候補:

- 探索終了
- 探索深度リセット
- 取得資源の一部ロスト

# 5. 戦闘システム

## 5.1 基本方針

- カード主体
- カードは自動で試行・発動される
- キャラごとにDeckを持つ
- プレイヤーは戦闘中に操作しない
- 通常攻撃を主役にしない
- 距離・座標・射程の詳細管理は行わない
- 複雑なAI判断は使用しない
- BattleEngineを戦闘状態と行動解決の唯一の正とする
- ViewerはBattleEngineが生成したEventとSnapshotを表示するだけとする

## 5.2 Action方式

戦闘は実時間ではなく、Action単位で進行する。

Actionとは、現在行動者へ制御が渡ってから、その行動結果と戦闘終了判定が確定するまでの最小進行単位である。

```text
Action開始
↓
現在行動者の確定
↓
ゲージ更新
↓
カード選択・試行
↓
Effect解決
↓
死亡判定
↓
戦闘終了判定
↓
Action終了
↓
次の生存キャラクターへ移行
```

Action IndexはBattle内の通し番号である。

```text
Action #1
Action #2
Action #3
```

表示上の演出時間はViewer側で管理する。

演出速度を変更しても、戦闘結果は変化しない。

## 5.3 M1の目的

M1では、1レーンに配置された固定の味方1体と敵1体による戦闘を対象とする。

M1の目的は、最終的な戦闘バランスを完成させることではなく、以下を検証可能にすることである。

- 現在行動者が常に一人だけ決まっている
- 1Actionずつ状態を更新できる
- カードを自動で試行・発動できる
- HPとMPが変化する
- Actionごとの構造化Eventを生成できる
- 死亡と戦闘終了を判定できる
- 同じ初期状態とSeedから同じ結果を再現できる
- ViewerでAction単位に再生できる

## 5.4 M1の対象外

M1では以下を実装しない。

- 3レーン
- SUPPORT
- 進軍
- ネクサス
- 速度ステータスによる行動順計算
- 追加Action
- 追撃
- 反撃
- 割り込み
- 同時行動
- 状態異常による行動不能
- 復活
- CardInstanceの性能ロール
- 装備
- PvE探索
- PvP

## 5.5 M1の戦闘開始状態

M1のBattleScenarioは、固定された以下のデータを入力として持つ。

- 味方CharacterMaster ID
- 敵CharacterMaster ID
- 味方Deck
- 敵Deck
- 固定行動順
- Battle Seed
- 最大Action数
- 各参加者の初期HP / MP
- 各参加者のDS / MPR / HPR

M1ではキャラクター配置、前衛・後衛、距離、射程を扱わない。

HP、MP、DS、MPR、HPRなどの初期値はCharacterMasterおよびBattleScenarioから生成する。

装備、育成値、スキルツリーはM1の入力に含めない。

## 5.6 M1の行動順

M1では行動順をBattleScenarioの固定配列として与える。

例:

```text
character_ally_001
character_enemy_001
```

BattleStateは内部的に以下を保持する。

- `turn_order`
- `current_turn_index`
- `action_index`
- 参加者ごとのRuntime State

Snapshotでは、表示上の行動者を以下の2項目へ分離する。

- `acted_actor_id`: そのActionで行動したキャラクター
- `next_actor_id`: 次に行動予定の生存キャラクター

戦闘開始時の現在行動者は`turn_order[0]`とする。

Action完了後は、配列上の次の生存キャラクターを次行動者とする。

配列の末尾へ到達した場合は先頭へ戻る。

M1では以下を行わない。

- 速度値による順番計算
- 行動速度ゲージ
- 同値判定
- ランダムな先攻決定
- 同一キャラクターの連続行動

固定行動順はM1専用の暫定仕様であり、3レーン対応前のAction Scheduler設計で再検討する。

## 5.7 M1における1Actionの境界

1Actionには以下を含む。

1. 次の生存キャラクターを現在行動者として確定する
2. `action_index`を1進める
3. `action_started`を生成する
4. 現在行動者のHPRを解決する
5. 現在行動者のMPRを解決する
6. 現在行動者のDSをDraw Gaugeへ加算する
7. Draw Gaugeがしきい値へ到達した回数分だけドローを試行する
8. 現在行動者が使用を試みるカードを手札先頭から決定する
9. MP、対象、発動条件を検証する
10. 発動可能な場合はMPを消費し、`card_used`を生成する
11. カードに定義されたEffectを記載順に解決する
12. 各Effect解決後に死亡判定を行う
13. カード内のEffect解決後に戦闘終了判定を行う
14. 戦闘継続時のみ次の生存キャラクターを決定する
15. `action_completed`を生成する
16. 勝敗確定時は`battle_completed`を生成する
17. Action完了後Snapshotを保存する

1枚のカードに複数Effectがある場合、それらを同一Action内で記載順にすべて解決する。

M1ではAction途中に別キャラクターのActionを割り込ませない。

## 5.8 M1のHPR / MPR / DS

HPR、MPR、DSはいずれも行動権を獲得した現在行動者にだけ適用する。

相手のAction中、行動権を持っていない側はHP回復、MP回復、Draw Gauge進行を行わない。

HPRとMPRはGaugeではなく、行動権獲得時の確定回復である。

```text
Action開始
↓
現在行動者のHPRをHPへ加算
↓
現在行動者のMPRをMPへ加算
↓
現在行動者のDSをDraw Gaugeへ加算
↓
しきい値到達回数分だけドロー試行
↓
現在行動者のカード処理
```

HPとMPは最大値を超えない。

```text
hp = min(max_hp, hp + hpr)
mp = min(max_mp, mp + mpr)
draw_gauge += ds
```

Draw Gaugeのしきい値は100とする。

しきい値以上になった場合は到達回数分だけドロー権を得て、超過分を繰り越す。

```text
trigger_count = draw_gauge // 100
draw_gauge = draw_gauge % 100
```

1回の加算で複数回しきい値へ到達した場合は、到達回数分だけ処理する。

死亡キャラクターは行動権を得ないため、HPR、MPR、DSを処理しない。

## 5.9 M1のカード選択

M1ではプレイヤーによるカード選択を行わない。

現在行動者は、手札を先頭から確認し、最初に発動可能なカードを使用する。

発動可能条件:

- 必要MP以上を持つ
- 生存している有効対象が存在する
- カード固有の追加条件を満たす

発動可能なカードがない場合、そのActionではカードを発動しない。

MP不足カードは捨てず、手札に保持する。

MP不足などで発動できないカードは`card_held`として記録する。

M1ではカードの優先度AI、評価関数、ランダム選択を実装しない。

## 5.10 M1の対象選択

M1は味方1体・敵1体のため、敵対象カードは生存している唯一の敵を対象とする。

味方対象カードは使用者自身を対象とする。

対象が存在しない場合、そのカードは発動できない。

対象選択に乱数は使用しない。

## 5.11 M1のダメージ

M1では、Damage Effectに定義された固定値をそのままHPから減算する。

```text
damage = effect.value
remaining_hp = max(0, current_hp - damage)
```

`damage_applied`は、解決前後を比較できるよう以下をPayloadへ含める。

- `before`
- `requested`
- `applied`
- `after`

M1では以下を計算しない。

- AD / AP倍率
- AR / MR軽減
- CRT
- EVA
- 属性相性
- ダメージ乱数
- 最低ダメージ補正

本番用ダメージ式は後続マイルストーンで定義する。

## 5.12 行動不能・死亡時の扱い

M1では状態異常による行動不能を実装しない。

死亡条件:

- HPが0になったキャラクターを死亡状態とする
- 死亡キャラクターはカードを使用できない
- 死亡キャラクターはカード対象にならない
- 死亡キャラクターは行動権を得ない
- 死亡キャラクターの順番へ到達した場合はAction Indexを消費せずスキップする

M1では復活を実装しない。

1対1戦闘のため、一方が死亡した時点で戦闘終了条件を満たす。

## 5.13 M1の戦闘終了条件

各Effect解決後に死亡判定を行う。

現在のカードに含まれるすべてのEffectを解決した後、以下の順で戦闘終了を判定する。

1. 味方と敵の双方が全滅している場合は引き分け
2. 敵が全滅している場合は味方勝利
3. 味方が全滅している場合は味方敗北
4. 最大Action数へ到達した場合は引き分け
5. 上記以外は戦闘継続

勝敗確定後:

- BattleStatusを`completed`へ変更する
- BattleResultを設定する
- `battle_completed`を生成する
- 次の現在行動者を決定しない
- 新しいActionを開始しない

M1の最大Action数はBattleScenarioで指定し、初期値は1000とする。

## 5.14 BattleStateの正

BattleStateの唯一の正はPythonのBattleEngineとする。

FastAPIはBattleEngineを呼び出すAPI層であり、戦闘ルールを保持しない。

ViewerはBattleStateを計算・変更しない。

M1ではBattleEngineが戦闘開始から終了までを一括計算し、以下を生成する。

- `events`
- `snapshots`
- `summary`
- `display_catalog`

`display_catalog`はViewer表示用の名前・説明を提供する補助情報であり、戦闘計算の正ではない。

BattleEngineは表示言語やMaster名称を保持しない。

API層はMasterデータから`display_catalog`を生成し、未登録IDは通常画面に露出させず`名称未設定`として扱う。

「次の行動を見る」は、Backendへ次のAction実行を要求する操作ではない。

Viewerが、計算済みReplayの表示位置を1Action進める操作とする。

M1ではBattleStateをDBまたはサーバーメモリへ永続保存しない。

ページ再読み込み時の復元、複数タブ操作、APIの二重送信制御はM1の対象外とする。

## 5.15 BattleEventの最小形式

画面表示用文字列ではなく、構造化されたBattleEventを正とする。

すべてのBattleEventは以下を持つ。

```json
{
  "event_id": 1,
  "action_index": 1,
  "sequence": 1,
  "event_type": "action_started",
  "actor_id": "ally_001",
  "target_id": null,
  "payload": {}
}
```

必須項目:

- `event_id`: Battle内のイベント通し番号
- `action_index`: 所属するAction Index。Battle開始イベントは0
- `sequence`: 同一Action内の表示順
- `event_type`: イベント種別
- `actor_id`: 発生主体。存在しない場合はnull
- `target_id`: 対象。存在しない場合はnull
- `payload`: イベント固有データ

M1で使用するEvent Type:

- `battle_started`
- `action_started`
- `gauge_changed`
- `card_drawn`
- `card_draw_blocked`
- `card_attempted`
- `card_held`
- `card_used`
- `mana_recovered`
- `mana_gained`
- `mana_spent`
- `health_recovered`
- `damage_applied`
- `character_defeated`
- `action_completed`
- `battle_completed`

Action権によるMPRは`mana_recovered`、カードEffectによるMP獲得は`mana_gained`として分離する。

`card_drawn`と`card_draw_blocked`は、Draw Gauge由来かカードEffect由来かを判別するため`draw_source`をPayloadへ持つ。

`card_draw_blocked`は以下をPayloadへ持つ。

- `blocked_reason`: `hand_full`または`empty_deck`
- `draw_source`: `draw_gauge`または`card_effect`
- `hand_size`
- `hand_limit`: `hand_full`時のみ

表示用の日本語コメントはViewer側でEventと`display_catalog`から生成する。

## 5.16 BattleSnapshotの最小形式

Viewerが任意のAction状態を表示できるよう、M1ではAction完了後のBattleStateをSnapshotとして保存する。

最低限保持する項目:

- `action_index`
- `battle_status`
- `battle_result`
- `acted_actor_id`
- `next_actor_id`
- 各参加者の`participant_id`
- 各参加者の`character_master_id`
- 各参加者の`side`
- 各参加者のHP / 最大HP
- 各参加者のMP / 最大MP
- 各参加者の生存状態
- 各参加者のDS / MPR / HPR
- 各参加者のDraw Gauge
- 各参加者の手札
- 各参加者の山札
- 各参加者の捨て札

Action #0として戦闘開始直前の初期Snapshotも保存する。

## 5.17 開発Viewerの操作

開発Viewerでは計算済みReplayをAction単位で確認する。

- 次の行動
- 前の行動
- 10行動進む
- 100行動進む
- 最後まで進む
- 最初へ戻る
- 任意Actionへ移動
- 自動再生
- 表示速度変更

「次の行動」を1回押すごとに、表示位置を1Actionだけ進める。

演出中は操作ボタンを無効化し、多重表示処理を防止する。

戦闘終了後は「次の行動」を無効化する。

表示速度と演出時間は戦闘計算へ影響しない。

M2以降の開発Viewerは、Action Summaryを以下のフェーズで表示する。

1. 行動準備
2. ドロー
3. カードアクション
4. 効果解決
5. 行動終了

通常表示では日本語名を優先し、内部IDや生EventはDebug表示に隔離する。

## 5.18 M1の再現性

M1では、同一の以下を入力した場合、同一のEvent、Snapshot、Summaryを生成する。

- `game-data.json`
- BattleScenario
- Deck
- Seed

M1のカード選択、対象選択、固定ダメージには乱数を使用しない。

山札シャッフルをM1で実施する場合は、必ずBattle Seedから生成したBattle専用乱数器を使用する。

# 13. カード設計

## 13.1 CardMaster

カードそのものの定義。

- id
- name
- rarity
- MPコスト
- Effect
- RollRule
- DeckRule

## 13.2 CardInstance

プレイヤーが実際に取得したカード個体。

- instanceId
- cardMasterId
- rolledValues
- 所有状態
- ロック状態
- お気に入り状態

## 13.3 レアリティ

レアリティはCardMaster自体の希少性を表す。

Legendaryは、単にMPが重いカードではなく、コスト効率を含めて意図的に強いカードとして設計可能とする。

## 13.4 性能ロール

同じCardMasterでも、取得したCardInstanceごとに性能差を持たせる。

レアリティと性能ロールは別概念とする。

## 13.5 RollRule

- fixed
- flat_range
- percent_range
- weighted_table

## 14.4 リソース

- DS: Draw Speed。行動権獲得時に現在行動者のDraw Gaugeへ加算する
- MPR: Mana Point Recovery。行動権獲得時に現在行動者のMPを確定回復する

MPRとHPRはGaugeではない。

DSのみ蓄積型Gaugeとして扱い、しきい値100へ到達した回数分だけドロー権を得る。

相手のAction中、行動権を持っていない側のDS / MPR / HPRは処理しない。

# 17. 現時点の重要結論

- 本作はカード主体である
- 通常攻撃主体にはしない
- 距離・座標管理はしない
- プレイヤーのリアルタイム操作は行わない
- 戦闘はAction Indexで進行する
- 表示時間は戦闘結果へ影響しない
- HPR / MPRは行動権獲得時の確定回復であり、DSのみDraw Gaugeを進める
- BattleStateの唯一の正はPythonのBattleEngineとする
- Viewerは戦闘計算を行わない
- BattleEngineは戦闘終了までを一括計算する
- 「次の行動を見る」は計算済みReplayの表示位置を1Action進める操作である
- M1では味方1体・敵1体の固定戦闘を扱う
- M1の行動順はBattleScenarioの固定配列とする
- M1では速度計算、反撃、追撃、割り込み、状態異常、復活を扱わない
- M1では固定ダメージを使用する
- M1では死亡または最大Action数で戦闘を終了する
- BattleEventは構造化データを正とし、表示文はViewer側で生成する
- 同一入力・同一Seedから同一結果を生成する
- M1はDefinition of Readyを満たしている
- レーン勝利という固定概念は設けない
- 進軍はレーン支配度である
- Support専用キャラは作らない
- CardMasterのレアリティとCardInstanceの性能ロールを分離する
- Legendaryは意図的に高効率な強カードとして設計可能とする
- Deck構築制限はデータ定義可能にする
