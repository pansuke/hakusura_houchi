# 2. 設計原則

1. BattleEngineとViewerを分離する
2. BattleEngineはFastAPIへ依存しない
3. Viewerは戦闘計算を行わない
4. 戦闘はAction Indexで進行する
5. 同一入力から同一結果を生成する
6. Masterデータは1レコード1JSONを正本とする
7. JSON SchemaをMaster構造の正本とする
8. PydanticモデルとJSON Schemaを手作業で二重管理しない
9. Master / Instance / Configuration / BattleStateを分離する
10. 生成物は再現可能にする
11. lower snake_caseをJSON・Event・Python属性名の標準とする
12. 未対応のruntime機能を暗黙に無視しない

# 6. CharacterMaster

## 6.1 M1形式

```json
{
  "id": "character_warrior_001",
  "name": "Warrior",
  "description": "M1 validation character.",
  "trait_ids": ["trait_none"],
  "resources": {
    "max_hp": 100,
    "max_mp": 10,
    "initial_mp": 0
  },
  "action_resources": {
    "ds": 25,
    "mpr": 1,
    "hpr": 0
  }
}
```

HPR / MPRはGaugeではない。DSだけがDraw Gaugeへ加算される。

## 6.2 M1フィールド

| JSON Path | 型 | 必須 | 制約 |
|---|---|---:|---|
| `$.id` | string | Yes | `^character_[a-z0-9_]+$` |
| `$.name` | string | Yes | 1～64文字 |
| `$.description` | string | Yes | 0～500文字 |
| `$.trait_ids` | string[] | Yes | 0～16件、重複禁止 |
| `$.resources.max_hp` | integer | Yes | 1～1000000 |
| `$.resources.max_mp` | integer | Yes | 0～1000000 |
| `$.resources.initial_mp` | integer | Yes | 0～max_mp |
| `$.action_resources.ds` | integer | Yes | 0～1000000 |
| `$.action_resources.mpr` | integer | Yes | 0～1000000 |
| `$.action_resources.hpr` | integer | Yes | 0～1000000 |

## 6.3 M3 CharacterMaster Schema v2

M3-Aで以下を追加する。

```json
{
  "combat_stats": {
    "ad": 80,
    "ap": 20,
    "ar": 30,
    "mr": 20,
    "push": 25
  }
}
```

| JSON Path | 型 | 制約 |
|---|---|---|
| `$.combat_stats.ad` | integer | 0以上 |
| `$.combat_stats.ap` | integer | 0以上 |
| `$.combat_stats.ar` | integer | M3では0以上 |
| `$.combat_stats.mr` | integer | M3では0以上 |
| `$.combat_stats.push` | integer | 0以上。50前後を高値の暫定目安とする |

CRT / EVA / Status ResistanceはM4-Bで追加する。

# 10. RollRule Schema

## 10.1 Binding

```json
{
  "target_path": "/effects/0/value",
  "rule": {
    "type": "flat_range",
    "min": 20,
    "max": 30,
    "step": 1
  }
}
```

- `target_path`は`^/effects/[0-9]+/value$`
- 対象Effect indexが実在することを整合性チェックする
- 対象値がintegerであることを整合性チェックする
- M1はRollRuleを使用せず、Effectの`value`を使用する
- RollRuleはM5のCardInstance生成で使用する

## 10.2 fixed

```json
{
  "type": "fixed",
  "value": 30
}
```

- valueはinteger

## 10.3 flat_range

```json
{
  "type": "flat_range",
  "min": 20,
  "max": 30,
  "step": 1
}
```

- min / max / stepはinteger
- min <= max
- step >= 1
- `(max - min)`がstepで割り切れなくてもよい
- minからstep刻みでmax以下の候補を生成する

## 10.4 percent_range

```json
{
  "type": "percent_range",
  "min_basis_points": -2500,
  "max_basis_points": 2500,
  "step_basis_points": 100
}
```

- 10000 basis points = 100%
- 値はinteger
- min_basis_points <= max_basis_points
- step_basis_points >= 1
- M5では対象の基礎値に対して適用し、最終値をintegerへ丸める
- 丸め方式はM5開始前に定義する

## 10.5 weighted_table

```json
{
  "type": "weighted_table",
  "entries": [
    {
      "value": 1,
      "weight": 80
    },
    {
      "value": 2,
      "weight": 18
    },
    {
      "value": 3,
      "weight": 2
    }
  ]
}
```

- entriesは1～100件
- valueはinteger
- weightは1～1000000のinteger
- weight合計を100へ揃える必要はない
- 同一valueの重複は禁止
- entries配列順を累積抽選順として保持する

# 19. Action Right Resource契約

現在行動者だけに以下を順番に適用する。

1. HPRをHPへ加算
2. MPRをMPへ加算
3. DSをDraw Gaugeへ加算

5. Gauge Eventを記録
6. 発火回数分だけDrawを試行

```text

mp = min(max_mp, mp + mpr)

```

# 24. M3 Action Scheduler

Battle全体で単一Action Indexを持つ。

固定順:

```text

top/enemy
mid/ally
mid/enemy
bot/ally
```

Participantが死亡中でもSlotを削除しない。

死亡中Slotでは:

- Action Indexを消費
- 復活待ちを更新
- 復活しなければ`respawn_waited`を生成して終了

```text
lane_id: top | mid | bot | null
```

Battle開始・全体終了Eventはnullを許可する。

# 25. M3 Deck Runtime

## 25.1 State

```text
DeckRuntimeState
- original_deck
- draw_pile
- hand
- discard_pile
- shuffle_count
```

M3-AまではCardMaster ID配列でよい。CardInstance IDへの移行はM5 / M6で行う。

## 25.2 初期化

1. original_deckをBattle専用乱数でshuffle
2. 先頭3枚をhand
3. 残りをdraw_pile
4. discard_pileは空

## 25.3 Draw

手札が7枚未満:

1. draw_pileから1枚取得
2. hand末尾へ追加

手札が7枚:

1. hand先頭をdiscard末尾へ移動
2. draw_pileから1枚取得
3. hand末尾へ追加
4. `card_overflow_discarded`を記録

## 25.4 Recycle

Draw前にdraw_pileが空でdiscardが存在する場合:

1. discardを決定的shuffle
2. draw_pileへ設定
3. discardを空にする
4. `discard_recycled`を記録

## 25.5 使用判定

hand先頭だけを判定する。

```text
oldest = hand[0]
```

使用不能なら:

- handに保持
- `card_held`
- 後続を確認しない
- Card Action終了

使用可能なら:

- hand先頭から削除
- MP消費
- Effect解決
- discard末尾へ追加

## 25.6 Action消費

CardMasterへ追加:

```json
{
  "grant_card_play": true
}
```

- true: 使用後終了
- false: 新しいhand先頭を続けて判定

技術的無限ループ防止は`simulation_safety_limit`で検出し、Battle Resultには変換しない。

# 27. Damage Calculator

Basis Pointsで中間値を保持する。

```text
raw_bp = base_damage * 10000
       + Σ(stat_value * ratio_bp)
```

physical:

```text
final = floor(raw_bp * defense_constant
              / (10000 * (defense_constant + AR)))
```

magic:

```text
final = floor(raw_bp * defense_constant
              / (10000 * (defense_constant + MR)))
```

true:

```text
final = floor(raw_bp / 10000)
```

Effectが有効に成立した場合:

```text
final_damage = max(minimum_damage, final)
```

HP適用時は対象の現在HPを超えた分を`requested`と`applied`で区別する。

```text
enemy_nexus = 1000
initial = 0
```

Participant Runtime:

```text
lane_id

```

## 28.2 非対面移動

行動者を敵Nexus方向へ`push`移動する。

ただし最寄りの生存ローカル敵位置または敵Nexusを越えない。

敵位置へ到達した場合は同位置へclampし、`engagement_started`を生成する。

## 28.3 対面PUSH

```text
advance = max(0, actor.push - opponent.push)
```

advance > 0なら、対面中の両Participantを行動者の進軍方向へ同量移動する。

相手死亡時はEngagementを解除し、次回以降は非対面移動へ移る。

## 28.4 行動可能条件

カード行動可能:

- ローカル敵と対面中
- または敵Nexus位置へ到達済み

それ以外は移動だけでActionを終了する。

# 29. Respawn Runtime

Participant Runtime:

```text
alive

```

死亡時:

```text
alive = false
respawn_turns_remaining = 3

```

```text
remaining > 0
→ remaining -= 1
→ Action終了
```

remainingが0の状態で次の自分のSlotが来た場合:

1. alive=true
2. HP=max_hp
3. MP=max_mp
4. position=own_nexus_position
5. Deck Runtime再初期化
6. Draw Gauge=0
7. 一時Effect解除
8. 移動・対面判定へ進む

復活は自身のAction開始時だけに発生する。

# 30. Target Resolver

## 30.1 Local

Enemy Effect:

1. 同レーンのengaged enemyがいればCharacter
2. いない場合、actorが敵Nexus位置へ到達していればNexus
3. それ以外は対象なし

Nexus対象を許可するのはM3ではDamage Effectだけ。

## 30.2 Adjacent

- TOP → MID
- BOT → MID
- MID → TOP / BOT

生存している敵Characterだけを候補にする。

MIDで複数候補がある場合は`adjacent_target_selection`乱数ストリームで1体選択する。

候補0件なら`no_valid_adjacent_target`。Nexusへfallbackしない。

## 30.3 Global

TOP → MID → BOTの順で生存敵Character全員を返す。

候補0件なら`no_valid_global_target`。

Nexusは絶対に候補へ含めない。

- ar: 0

- ally Nexus HP <= 0: ally_loss
